# Decisions

Important architectural, technical, and process decisions along with their reasoning.

## Format

```
[YYYY-MM-DD] Decision: <what was decided>
Reason: <why this option was chosen>
Alternatives: <other options considered and why they were rejected>
Status: active | superseded | revisit
```

## When to Read
- Start of every session — understand prior decisions
- Before proposing changes to architecture, tooling, or process
- When a user asks "why do we…" or "why did we choose…"

## When to Write
- When the user explicitly decides between alternatives
- When an architectural or tooling choice is made
- Update `Status` to `superseded` if a newer decision replaces it — keep the old entry for history
- Check existing entries before adding — don't duplicate
