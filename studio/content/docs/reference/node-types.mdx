title: Node Types
description: Step and sub-workflow specifications — the two node types in AgentFlow.

Every node in an AgentFlow workflow is one of two types: **step** or **sub-workflow**. Each is defined as a `SKILL.md` file inside a named directory. Nodes with conditional edges are automatically rendered as diamond-shaped gateways (routers) on the canvas.

<Tabs items={["Step", "Sub-workflow"]}>
<Tab value="Step">

A step is the primary execution unit. It runs instructions, invokes tools, and produces outputs. This is the default node type.

## Behavior

- Executes sequentially within the workflow graph
- Can reference capabilities (tools), instructions, and skills
- Supports context inputs from other nodes via data flow edges
- Produces outputs declared in frontmatter
- When a step has conditional edges, it is automatically rendered as a gateway (router) on the canvas

## Router inference

Steps with conditional edges automatically render as diamond-shaped gateways on the canvas. You don't need a separate router type — just add conditional edges:

```markdown
# review-design-gate/SKILL.md
---
name: review-design-gate
type: step
---

# Review Design Gate

Present the design from {{<< output.create-design}} to the user.

{{-> nodes/plan-tasks | design approved by user}}
{{-> nodes/create-design | design rejected or changes requested}}
```

This step has two conditional edges, so it renders as a gateway. Common patterns:
- Approval gates (present work, ask for approval, route based on response)
- Completion checks (are all tasks done? route to finish or loop back)
- Triage decisions (classify input, route to the appropriate handler)

## Applicable validation rules

- `no_entry_point` — at least one step must have `entry: true`
- `broken_ref` — all resource references must resolve
- `broken_data_flow` — input refs must point to existing nodes
- `context_budget` / `context_budget_high` — token limits
- `output_declaration` — outputs referenced by other nodes must be declared

<Tab value="Sub-workflow">

A sub-workflow embeds another workflow as a single node, enabling composition and reuse.

## Behavior

- References an entire workflow directory
- Executes the referenced workflow as a nested graph
- Inputs are passed into the sub-workflow entry node
- Outputs are collected from the sub-workflow terminal nodes

## Applicable validation rules

- `broken_ref` — the workflow path must resolve to a valid workflow
- `cycle` — sub-workflow must not create circular references
- `broken_data_flow` — input/output refs must be valid
- `unreachable` — the nested workflow must itself be valid
