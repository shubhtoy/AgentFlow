Resources are the reusable building blocks that nodes pull into their context via `{{category/name}}`. Each category serves a different purpose and has different rules about when to use it.

## When to Use What

| You want to... | Use | Example |
|----------------|-----|---------|
| Set coding conventions | **Instruction** | `{{instructions/code-style}}` |
| Give the agent a tool | **Capability** | `{{capabilities/write-file}}` |
| Define a routing condition | **Runbook** (condition) | `{{runbooks/tests-pass}}` |
| Add a human checkpoint | **Runbook** (approval) | `{{runbooks/review-design}}` |
| Persist decisions across runs | **Memory** | `{{memory/decisions}}` |
| Automate on file changes | **Hook** | `hooks/lint-on-save.json` |

## Instructions

**What:** Guidance the agent treats as constraints — conventions, strategies, best practices.

**When:** When multiple nodes need the same behavioral rules. Don't create an instruction for something only one node uses — put it in the SKILL.md body instead.

**Real example** from the library (`implementation-discipline.md`):

```yaml
---
name: implementation-discipline
type: instruction
scope: workflow
domain: development
description: Best practices for writing production-quality code
tags: [coding, best-practices, quality]
---

# Implementation Discipline

## Before Writing Code
1. Re-read the task description and the relevant design section
2. Check {{memory/decisions}} for any relevant past decisions
3. Identify the minimal change needed — don't over-engineer

## While Writing Code
1. Follow existing code conventions (naming, formatting, structure)
2. Write the smallest possible change that satisfies the task
3. Run {{capabilities/get-diagnostics}} after every file edit
4. Run {{capabilities/run-tests}} after every logical change
```

Notice how the instruction itself references other resources — capabilities and memory. When a node loads this instruction, those nested references are also resolved.

## Capabilities

**What:** Tool definitions that tell the agent what it can do and how to invoke it.

**When:** Whenever a node needs to perform an action — read files, run tests, call an API, execute a script.

**Four types:**

<Tabs items={['builtin', 'script', 'mcp', 'package']}>
  <Tab value="builtin">
    Maps to a built-in tool the agent platform provides:

    ```yaml
    ---
    name: write-file
    type: builtin
    builtin_mapping: fsWrite
    description: Create or modify files.
    outputs: [file_path, bytes_written]
    narrativeTemplate:
      prefix: "Use"
      suffix: "to create or modify the file"
    ---

    # Write File

    Create or modify files. Supports creating new files,
    replacing content, and appending to existing files.
    ```

    The `narrativeTemplate` generates natural language when the capability is referenced: "Use **write-file** to create or modify the file."
  </Tab>
  <Tab value="script">
    Runs a shell command:

    ```yaml
    ---
    name: run-tests
    type: script
    command: "npm test -- --related"
    description: Run related tests for modified files
    ---

    Runs the test suite filtered to tests related to recently
    modified files. Returns pass/fail status and error output.
    ```
  </Tab>
  <Tab value="mcp">
    Calls a tool on an MCP server (see [MCP](/docs/authoring/mcp)):

    ```yaml
    ---
    name: create-issue
    type: mcp
    mcp: github-server
    description: Create a new GitHub issue
    parameters:
      title: { type: string, required: true }
      body: { type: string }
    generated: true
    ---
    ```

    These are usually auto-generated by `agentflow mcp discover`.
  </Tab>
  <Tab value="package">
    References an installable package:

    ```yaml
    ---
    name: agent-browser
    type: package
    package: agent-browser
    description: Browser automation CLI for AI agents
    ---

    Install: `npm i -g agent-browser && agent-browser install`
    ```
  </Tab>
</Tabs>

## Runbooks

**What:** Decision criteria for routers and human interaction protocols.

**When:** Every conditional edge needs a runbook. Every human checkpoint needs one.

**Two main types:**

<Tabs items={['Conditions', 'Interactions']}>
  <Tab value="Conditions">
    Used by router nodes to decide which edge to take:

    ```yaml
    ---
    type: condition
    name: tests-pass
    check: All tests pass and no diagnostics errors remain
    ---

    # Tests Pass

    **True when:** All tests pass, no lint errors, no type errors.
    **False when:** Any test fails or diagnostics report errors.
    ```

    The `check` field is a one-line summary the agent evaluates. The body provides detailed criteria.
  </Tab>
  <Tab value="Interactions">
    Define human-agent interaction protocols:

    ```yaml
    ---
    type: approval
    name: review-design
    timeout: 300
    ---

    # Review Design

    Present the design document to the user.

    **Show:** Architecture overview, key decisions, trade-offs.
    **Options:** Approve, Reject with feedback, Request changes.
    ```

    Types: `approval` (present + approve/reject), `confirm` (yes/no), `escalation` (hand to human).
  </Tab>
</Tabs>

## Memory

**What:** Persistent state that survives across workflow runs. The only resource category the agent can write to.

**When:** For decisions, preferences, user context, and learnings that should accumulate over time.

```yaml
---
name: decisions
description: Architecture and technology decisions
editable: true
---

## 2024-03-15: Database Choice

Chose PostgreSQL over MongoDB for relational data integrity.
Trade-off: less flexible schema, but stronger consistency.

## 2024-03-20: State Management

Chose Zustand over Redux. Simpler API, less boilerplate.
```

Memory files grow over time. The agent appends new entries as it makes decisions. Use `scope: summary` in node frontmatter to load a condensed version when the full history would be too many tokens.

## Scoping Rules

| Level | Location | Visible to |
|-------|----------|-----------|
| **Global** | `.agentflow/instructions/` | All workflows, all nodes |
| **Workflow** | `.agentflow/build-feature/instructions/` | Only build-feature nodes |
| **Node** | `.agentflow/build-feature/implement/context.md` | Only the implement node |

Workflow-scoped resources override global ones with the same name. This lets you customize behavior per workflow without changing the global defaults.

## Platform Support

Not all resource types export the same way to every platform:

| Resource | Claude Code | Cursor | Kiro | Windsurf | VS Code |
|----------|------------|--------|------|----------|---------|
| Instructions | Inline | Rules | Steering | Rules | Instructions |
| Capabilities | Tools | Tools | Tools | Tools | Tools |
| Runbooks | Inline | Rules | Specs | Rules | Instructions |
| Memory | Context | Context | Context | Context | Context |
| Hooks | Native | As instructions | Native | As instructions | Native |

Platforms without native hook support get hooks converted to natural language instructions describing the automation behavior.
