## Reserved Directories

Five directory names have special meaning. The parser treats them as resource categories. Everything else is a workflow.

## Naming Conventions

### Directories and files

Use **kebab-case** for everything: `gather-requirements`, `code-style.md`, `test-on-change.json`. This isn't enforced by the validator, but the CLI and library follow it consistently.

### Special file names

The parser looks for files in this priority order when determining a node's primary file:

1. Any file with `primary: true` in frontmatter
2. `SKILL.md`
3. `main.md`
4. Alphabetically first `.md` file

<Callout type="info" title="AGENTS.md is required">
  Every workspace root and every workflow directory must have an `AGENTS.md` file with `type: agents` in its frontmatter. The validator emits an error if it's missing.
</Callout>

## Workflow Directories

Any subdirectory of `.agentflow/` that isn't a reserved directory is treated as a workflow. Inside a workflow:

<Tabs items={['Nodes', 'Workflow-scoped resources', 'Hooks']}>
  <Tab value="Nodes">
    Each subdirectory is a node. It must contain at least one `.md` file:

    ```
    build-feature/
      gather-requirements/
        SKILL.md              ← primary file
        requirements-template.md  ← supporting context (primary: false)
      create-design/
        SKILL.md
    ```

    Multi-file nodes: all `.md` files are parsed. One is primary (the task definition), the rest are loaded as additional context for that node.
  </Tab>
  <Tab value="Workflow-scoped resources">
    Workflows can have their own resource directories. These are only visible to nodes within that workflow:

    ```
    build-feature/
      instructions/
        requirements-elicitation.md   ← only build-feature nodes see this
      runbooks/
        design-approved.md            ← only build-feature nodes see this
    ```

    If a workflow-scoped resource has the same name as a global one, the workflow-scoped version takes precedence.
  </Tab>
  <Tab value="Hooks">
    Workflow-scoped hooks live in a `hooks/` directory inside the workflow:

    ```
    build-feature/
      hooks/
        test-on-change.json
        lint-on-save.json
    ```

    These hooks only fire when the workflow is active.
  </Tab>
</Tabs>

## Excluded Directories

The parser skips these directories entirely — they're never treated as nodes or resources:

```
node_modules, .git, dist, build, output, .next,
__pycache__, .venv, venv, .cache, coverage
```

The `output/` directory inside a node is for artifacts, not context.

## Config File

The optional `agentflow.config.json` lives in the **project root** (parent of `.agentflow/`), not inside the workspace:

```json
{
  "name": "MyTool",
  "dir": ".mytool",
  "cli": "mytool"
}
```

<Accordions>
  <Accordion title="Config fields">
    | Field | Type | Default | Description |
    |-------|------|---------|-------------|
    | `name` | string (1-64 chars) | `"AgentFlow"` | Brand name shown in UI |
    | `dir` | string (1-64 chars) | `".agentflow"` | Workspace directory name |
    | `cli` | string (1-32 chars) | `"agentflow"` | CLI command name |
    | `logo` | string | — | Path to logo file |
    | `theme` | string | — | Theme name |

    Precedence: env vars (`AGENTFLOW_BRAND_NAME`, `AGENTFLOW_DIR`, `AGENTFLOW_CLI`) > config file > defaults.
  </Accordion>
</Accordions>
