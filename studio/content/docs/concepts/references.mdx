References are the `{{...}}` expressions inside SKILL.md files. They serve two purposes: declaring what context a node needs (resource mentions, data flow) and declaring where execution goes next (edges).

The parser reads all references to build the workflow graph and determine each node's context dependencies.

## Resource Mentions

Resource mentions load a file from one of the five resource categories into the node's context window. The agent receives the full content of the referenced file as part of its instructions.

```markdown
{{instructions/code-style}}        → loads instructions/code-style.md
{{capabilities/write-file}}        → loads capabilities/write-file.md
{{runbooks/design-approved}}       → loads runbooks/design-approved.md
{{memory/decisions}}               → loads memory/decisions.md
```

### Resolution order

When the parser encounters a resource mention, it resolves the file in this order:

1. **Workflow-scoped** — check `build-feature/instructions/code-style.md` first
2. **Global** — fall back to `.agentflow/instructions/code-style.md`
3. **Error** — if neither exists, the validator emits `broken_ref`

This means a workflow can override a global resource by placing a file with the same name in its own directory. The closest file wins — the same pattern as the AGENTS.md standard's nested file precedence.

### What the agent actually sees

When you write `{{instructions/code-style}}` in a SKILL.md, the runtime replaces the reference with the full content of that file:

```markdown
## TypeScript Conventions

- Use strict mode
- Prefer functional patterns
- No `any` types — use `unknown` and narrow
- All exports must have explicit type annotations
```

The reference itself disappears — the agent sees the content, not the `{{...}}` syntax.

## Edge References

Edge references declare execution flow between nodes. Unlike resource mentions, they don't load content into context — they tell the parser how to build the directed graph.

### Unconditional edges

Always go to the target node after this one completes:

```markdown
{{-> nodes/create-design}}
```

A node can have multiple unconditional edges (fan-out), but this is uncommon. Most nodes have a single outgoing edge.

### Conditional edges

Go to the target node only if a condition is met. The condition is defined in a runbook file:

```markdown
{{-> nodes/plan-tasks | runbooks/design-approved}}
{{-> nodes/create-design | runbooks/design-rejected}}
```

The pipe `|` separates the target from the condition. The runbook must have `type: condition` in its frontmatter. Router nodes typically have multiple conditional edges — one for each possible outcome.

### Edge validation

The validator checks several edge-related rules:

- `missing_condition` — conditional edge references a runbook that doesn't exist
- `router_non_conditional` — router has unconditional edges (routers should always use conditions)
- `router_single_exit` — router with only one outgoing edge (probably a mistake)

## Data Flow References

Data flow references load another node's output into the current node's context. This is how information passes between steps without loading everything into a single prompt.

```markdown
Using the design from {{<< output.create-design}}, implement the feature.
```

### Requirements

The source node must declare its outputs in frontmatter:

```yaml
---
name: create-design
type: step
outputs:
  - name: design-doc
    format: markdown
    description: Technical design document
---
```

Data flow refs can only point backward in the graph — to nodes that execute before the current one. The validator emits `broken_data_flow` if you reference a node that doesn't exist or hasn't declared outputs.

### Scope control

You can control how much of a prior output is loaded using the `context.inputs` field in frontmatter:

```yaml
context:
  inputs:
    - ref: output.create-design
      scope: full       # load entire output
    - ref: output.gather-requirements
      scope: summary    # load condensed version
```

`full` loads everything. `summary` loads a condensed version — useful when you need awareness of a prior step's result without consuming your entire token budget.

## Template Variables

Template variables resolve workspace-level configuration at export time (not at runtime):

```markdown
{{$workflows}}     → list of all workflows in the workspace
{{$name}}          → workspace name from root AGENTS.md
{{$description}}   → workspace description
```

These are primarily used in the root `AGENTS.md` to generate dynamic content:

```markdown
# My Workspace

## Available Workflows

{{$workflows}}
```

The validator emits `malformed_variable` if the syntax is invalid.

## Workflow References

Workflow references link a sub-workflow node to its target workflow's `AGENTS.md`. When the parser encounters `{{workflows/name}}`, it resolves the path to the named workflow directory and loads its identity descriptor.

```markdown
{{workflows/ci-cd}}        → links to .agentflow/ci-cd/AGENTS.md
{{workflows/deploy}}       → links to .agentflow/deploy/AGENTS.md
```

This is the mechanism that connects a sub-workflow node in a parent workflow to the actual workflow it delegates to. The referenced workflow must exist in the workspace and have a valid `AGENTS.md`.

The validator emits `missing_sub_workflow_target` if the referenced workflow directory does not exist, and `unresolved_workflow_ref` if the `AGENTS.md` inside it is missing or invalid.

## Putting It All Together

Here's a complete SKILL.md showing all reference types working together:

```markdown
---
name: implement
type: step
outputs:
  - name: implementation
    format: code
---

# Implement the Feature

Based on the approved design in {{<< output.create-design}},
implement the feature following {{instructions/code-style}} conventions.

Use {{capabilities/write-file}} to create files and
{{capabilities/run-tests}} to verify correctness.

When complete:
{{-> nodes/verify-feature}}
```

This single file declares:
- **1 data flow ref** — loads the design output from the previous step
- **2 resource mentions** — loads code-style instructions and two capability definitions
- **1 unconditional edge** — flows to verify-feature when done

The parser extracts all four reference types to build the graph and determine this node's context window.

<Callout type="warn" title="Explicit is everything">
  AgentFlow never loads resources implicitly. If a file isn't referenced, it's not in the context. This means you always know exactly what a node will see — and you can verify it with the validator before running.
</Callout>
