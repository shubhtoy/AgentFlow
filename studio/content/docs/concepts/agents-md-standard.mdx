AgentFlow is built on [AGENTS.md](https://agents.md) — an open format for guiding coding agents, used by over 60,000 open-source projects and stewarded by the Agentic AI Foundation under the Linux Foundation.

One AGENTS.md file works across Claude Code, Cursor, Windsurf, VS Code Copilot, Kiro, Devin, Aider, Zed, Warp, and more. Write once, run everywhere.

## The Two File Types

<Tabs items={['AGENTS.md', 'SKILL.md']}>
  <Tab value="AGENTS.md">
    **Identity and routing** — who the agent is, what it knows, how it behaves.

    One per workspace (root identity) + one per workflow (workflow identity). Always loaded when active.

    ```yaml
    ---
    name: Build Feature
    description: Spec-driven feature development with approval gates.
    identity:
      role: Senior software engineer
      constraints:
        - Always write tests before implementation
        - Never modify files outside the project scope
    ---
    ```
  </Tab>
  <Tab value="SKILL.md">
    **Node contract** — what this specific step does, its inputs, outputs, and behavior.

    One per node. Loaded only when that node is executing.

    ```yaml
    ---
    name: gather-requirements
    type: step
    entry: true
    outputs:
      - name: requirements-doc
        format: markdown
        description: Structured requirements document
    ---

    # Gather Requirements

    Interview the user to understand what they want to build.
    Reference {{instructions/requirements-elicitation}} for technique.
    ```

    <Callout type="info" title="Origin">
      SKILL.md was introduced by Anthropic for Claude Code's agent skills system. See [Equipping Agents for the Real World with Agent Skills](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills) for the progressive disclosure pattern — frontmatter is always loaded, but the full skill body loads only when relevant.
    </Callout>
  </Tab>
</Tabs>

## Nested Files and Precedence

Like the standard, AgentFlow supports nested AGENTS.md files. The closest file to the current context wins:

<Files>
  <Folder name=".agentflow" defaultOpen>
    <File name="AGENTS.md" />
    <Folder name="build-feature" defaultOpen>
      <File name="AGENTS.md" />
      <Folder name="implement">
        <File name="SKILL.md" />
      </Folder>
    </Folder>
  </Folder>
</Files>

```
.agentflow/AGENTS.md              ← workspace identity (always loaded)
.agentflow/build-feature/AGENTS.md ← workflow identity (loaded when active)
.agentflow/build-feature/implement/SKILL.md ← node contract (loaded when executing)
```

Each level can override or extend the parent. This mirrors the monorepo pattern from the AGENTS.md spec.

## The Ecosystem in Practice

AGENTS.md is not just a file format — it is a universal interface between humans and AI coding agents. When you write an AGENTS.md file for AgentFlow, that same file communicates your intent to Claude Code, Cursor, Windsurf, VS Code Copilot, and any other tool that supports the standard.

This works because the standard is intentionally minimal at its base layer. Every platform reads the same markdown and extracts what it needs. AgentFlow's extensions (typed frontmatter, references, resource categories) are additive — platforms that don't understand them simply ignore the YAML fields and treat the body as plain instructions.

### How Each Platform Interprets the File

**Claude Code** natively understands both AGENTS.md and SKILL.md. It reads the identity section for personality and constraints, and loads skills on demand using the progressive disclosure pattern. AgentFlow's typed frontmatter fields are recognized directly.

**Cursor** treats AGENTS.md as a rules source. It extracts the markdown body and applies it as behavioral guidance. Cursor ignores YAML frontmatter it doesn't recognize but respects the prose instructions.

**VS Code Copilot** looks for instruction files in `.github/copilot/`. AgentFlow's export resolves references and flattens the multi-file structure into a single instructions file that Copilot understands.

**Generic platforms** that support AGENTS.md read the file as markdown documentation about how to behave in this repository. Even without typed parsing, the human-readable instructions provide value.

### The Write-Once Benefit

Because AGENTS.md is the source of truth, you maintain one set of agent instructions regardless of how many platforms your team uses. A developer using Claude Code and another using Cursor both get consistent behavior from the same file. When you update the instructions, every platform picks up the change on next read.

This eliminates the "configuration drift" problem where each developer's AI tool behaves differently because they configured it independently.

## AgentFlow's Extensions in Detail

The base AGENTS.md standard provides a blank canvas — any markdown content works. AgentFlow adds five structural extensions that enable validation, visual editing, and multi-platform export. Each extension exists for a specific reason.

### Typed Frontmatter (YAML)

AgentFlow requires YAML frontmatter with specific fields:

```yaml
---
name: gather-requirements
type: step
entry: true
outputs:
  - name: requirements-doc
    format: markdown
    description: Structured requirements document
context:
  max_tokens: 6000
---
```

**Why YAML?** Three reasons. First, YAML is already the standard frontmatter format for markdown files (used by Jekyll, Hugo, Docusaurus, MDX). Developers expect it. Second, YAML is both human-readable and machine-parseable — the validator can check types and required fields without custom parsing. Third, YAML separates structured metadata from freeform instructions cleanly. The frontmatter is for machines; the body is for the LLM.

**What it enables:** The typed fields let the validator check that outputs are declared before they're referenced, that entry nodes exist, that context budgets are numbers, and that node types are valid. Without typed frontmatter, validation would require natural language understanding of the markdown body.

### SKILL.md (Progressive Disclosure)

AgentFlow uses SKILL.md for node-level definitions, following the pattern Anthropic introduced for Claude Code's agent skills system.

The progressive disclosure pattern works like this: the frontmatter (name, type, outputs) is always available to the routing layer — it's cheap to load and tells the system what this node does. The full markdown body loads only when the node actually executes. This means the system can make routing decisions (which node to run next) without loading every node's full instructions into context.

This is analogous to how a function signature tells you what a function does without reading its implementation. The frontmatter is the signature; the body is the implementation.

### Resource Categories (Why Five)

AgentFlow defines five resource categories: instructions, capabilities, runbooks, memory, and hooks.

**Why exactly five?** The categories map to distinct roles in the context window:

- **instructions** — behavioral guidance the agent follows (how to write code, how to communicate)
- **capabilities** — tool definitions the agent can invoke (what actions are available)
- **runbooks** — conditional logic and decision criteria (when to route, when to approve)
- **memory** — persistent state across runs (learned preferences, accumulated knowledge)
- **hooks** — event triggers that fire on workflow events (on-complete, on-error)

Fewer categories would force unrelated content into the same bucket — tool definitions mixed with behavioral instructions confuse the model about what is an action versus a rule. More categories would create artificial distinctions that don't map to how LLMs process context. Five categories cover the fundamental types of information an agent needs without over-segmenting.

### Reference Syntax (Double-Braces)

AgentFlow uses `{{category/name}}` syntax for references:

```markdown
Follow {{instructions/code-style}} conventions.
Use {{capabilities/write-file}} to create files.
Check {{<< output.create-design}} for the approved design.
Route to {{-> nodes/implement}} when ready.
```

**Why double-braces?** Single braces conflict with JavaScript template literals, JSON, and many templating languages. Double-braces are visually distinct in markdown, unlikely to appear in normal prose, and already familiar from Mustache/Handlebars templating. They signal "this is a reference that the system resolves" without ambiguity.

**How the parser handles them:** The parser uses a single regex pass over the SKILL.md body to extract all `{{...}}` expressions. Each match is classified by its prefix:

- No prefix: resource mention (loads content into context)
- `->` prefix: edge declaration (builds graph structure)
- `<< output.` prefix: data flow (loads prior node output)
- `$` prefix: template variable (resolved at export time)

References are resolved in two passes: first checking workflow-scoped resources (files within the workflow directory), then falling back to global resources (files in the workspace-level category directories). This scoping lets workflows override global resources when needed.
