# Stakeholder Communications

Craft clear, audience-appropriate communications for different stakeholders.

## Process Steps
1. Identify the audience and their decision-making context
2. Choose the format: executive summary, status update, or escalation
3. Lead with the ask or key takeaway — not the backstory
4. Use numbers over adjectives ("3x slower" not "much slower")
5. Include exactly one action item per communication
6. Review for brevity — cut anything that doesn't serve the reader

## Executive Summary Format
1. **Status**: One line — Green/Yellow/Red
2. **Key Metric**: The number that matters most
3. **Decision Needed**: What you need from them (if anything)
4. **Timeline**: When it matters by
5. **Risk**: What could go wrong

## Status Update Format
- What was accomplished since last update
- What's planned next
- Blockers requiring help
- Metrics/progress against goals

## Escalation Format
1. What happened (facts only, 2-3 sentences)
2. Impact (who is affected, how badly)
3. What's been tried
4. What help is needed
5. Proposed next steps with timeline

## Anti-Patterns
- Burying the ask at the bottom of a long email
- Using technical jargon with non-technical stakeholders
- Sending status updates with no clear action item or takeaway
- Escalating without proposing a solution
- Over-communicating minor issues to senior leadership

## Output Format
A communication draft with: subject line, audience, format type (summary/update/escalation), body text, and explicit action item.

## Examples

### Example: Project status update
**Subject**: Sprint 4 Status — Yellow (blocked on API dependency)

**Accomplished**: Completed auth module and user dashboard (8/10 stories done).
**Next**: Integration testing with payments API.
**Blocker**: Payments API not yet deployed to staging — need DevOps help by Friday.
**Action**: @DevOps — please prioritize payments API staging deploy.
